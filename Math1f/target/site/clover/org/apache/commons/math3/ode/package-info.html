17    /**
18    *
19    * <p>
20    * This package provides classes to solve Ordinary Differential Equations problems.
21    * </p>
22    *
23    * <p>
24    * This package solves Initial Value Problems of the form
25    * <code>y'=f(t,y)</code> with <code>t<sub>0</sub></code> and
26    * <code>y(t<sub>0</sub>)=y<sub>0</sub></code> known. The provided
27    * integrators compute an estimate of <code>y(t)</code> from
28    * <code>t=t<sub>0</sub></code> to <code>t=t<sub>1</sub></code>.
29    * It is also possible to get thederivatives with respect to the initial state
30    * <code>dy(t)/dy(t<sub>0</sub>)</code> or the derivatives with
31    * respect to some ODE parameters <code>dy(t)/dp</code>.
32    * </p>
33    *
34    * <p>
35    * All integrators provide dense output. This means that besides
36    * computing the state vector at discrete times, they also provide a
37    * cheap mean to get the state between the time steps. They do so through
38    * classes extending the {@link
39    * org.apache.commons.math3.ode.sampling.StepInterpolator StepInterpolator}
40    * abstract class, which are made available to the user at the end of
41    * each step.
42    * </p>
43    *
44    * <p>
45    * All integrators handle multiple discrete events detection based on switching
46    * functions. This means that the integrator can be driven by user specified
47    * discrete events. The steps are shortened as needed to ensure the events occur
48    * at step boundaries (even if the integrator is a fixed-step
49    * integrator). When the events are triggered, integration can be stopped
50    * (this is called a G-stop facility), the state vector can be changed,
51    * or integration can simply go on. The latter case is useful to handle
52    * discontinuities in the differential equations gracefully and get
53    * accurate dense output even close to the discontinuity.
54    * </p>
55    *
56    * <p>
57    * The user should describe his problem in his own classes
58    * (<code>UserProblem</code> in the diagram below) which should implement
59    * the {@link org.apache.commons.math3.ode.FirstOrderDifferentialEquations
60    * FirstOrderDifferentialEquations} interface. Then he should pass it to
61    * the integrator he prefers among all the classes that implement the
62    * {@link org.apache.commons.math3.ode.FirstOrderIntegrator
63    * FirstOrderIntegrator} interface.
64    * </p>
65    *
66    * <p>
67    * The solution of the integration problem is provided by two means. The
68    * first one is aimed towards simple use: the state vector at the end of
69    * the integration process is copied in the <code>y</code> array of the
70    * {@link org.apache.commons.math3.ode.FirstOrderIntegrator#integrate
71    * FirstOrderIntegrator.integrate} method. The second one should be used
72    * when more in-depth information is needed throughout the integration
73    * process. The user can register an object implementing the {@link
74    * org.apache.commons.math3.ode.sampling.StepHandler StepHandler} interface or a
75    * {@link org.apache.commons.math3.ode.sampling.StepNormalizer StepNormalizer}
76    * object wrapping a user-specified object implementing the {@link
77    * org.apache.commons.math3.ode.sampling.FixedStepHandler FixedStepHandler}
78    * interface into the integrator before calling the {@link
79    * org.apache.commons.math3.ode.FirstOrderIntegrator#integrate
80    * FirstOrderIntegrator.integrate} method. The user object will be called
81    * appropriately during the integration process, allowing the user to
82    * process intermediate results. The default step handler does nothing.
83    * </p>
84    *
85    * <p>
86    * {@link org.apache.commons.math3.ode.ContinuousOutputModel
87    * ContinuousOutputModel} is a special-purpose step handler that is able
88    * to store all steps and to provide transparent access to any
89    * intermediate result once the integration is over. An important feature
90    * of this class is that it implements the <code>Serializable</code>
91    * interface. This means that a complete continuous model of the
92    * integrated function throughout the integration range can be serialized
93    * and reused later (if stored into a persistent medium like a filesystem
94    * or a database) or elsewhere (if sent to another application). Only the
95    * result of the integration is stored, there is no reference to the
96    * integrated problem by itself.
97    * </p>
98    *
99    * <p>
100    * Other default implementations of the {@link
101    * org.apache.commons.math3.ode.sampling.StepHandler StepHandler} interface are
102    * available for general needs ({@link
103    * org.apache.commons.math3.ode.sampling.DummyStepHandler DummyStepHandler}, {@link
104    * org.apache.commons.math3.ode.sampling.StepNormalizer StepNormalizer}) and custom
105    * implementations can be developed for specific needs. As an example,
106    * if an application is to be completely driven by the integration
107    * process, then most of the application code will be run inside a step
108    * handler specific to this application.
109    * </p>
110    *
111    * <p>
112    * Some integrators (the simple ones) use fixed steps that are set at
113    * creation time. The more efficient integrators use variable steps that
114    * are handled internally in order to control the integration error with
115    * respect to a specified accuracy (these integrators extend the {@link
116    * org.apache.commons.math3.ode.nonstiff.AdaptiveStepsizeIntegrator
117    * AdaptiveStepsizeIntegrator} abstract class). In this case, the step
118    * handler which is called after each successful step shows up the
119    * variable stepsize. The {@link
120    * org.apache.commons.math3.ode.sampling.StepNormalizer StepNormalizer} class can
121    * be used to convert the variable stepsize into a fixed stepsize that
122    * can be handled by classes implementing the {@link
123    * org.apache.commons.math3.ode.sampling.FixedStepHandler FixedStepHandler}
124    * interface. Adaptive stepsize integrators can automatically compute the
125    * initial stepsize by themselves, however the user can specify it if he
126    * prefers to retain full control over the integration or if the
127    * automatic guess is wrong.
128    * </p>
129    *

132    * <tr BGCOLOR="#CCCCFF"><td colspan=2><font size="+2">Fixed Step Integrators</font></td></tr>

139    * </table>
140    * </p>

150    * <tr><td>{@link org.apache.commons.math3.ode.nonstiff.AdamsMoultonIntegrator Adams-Moulton}</td><td>variable</td><td>variable</td></tr>
151    * </table>
152    * </p>
153    *
154    * <p>
155    * In the table above, the {@link org.apache.commons.math3.ode.nonstiff.AdamsBashforthIntegrator
156    * Adams-Bashforth} and {@link org.apache.commons.math3.ode.nonstiff.AdamsMoultonIntegrator
157    * Adams-Moulton} integrators appear as variable-step ones. This is an experimental extension
158    * to the classical algorithms using the Nordsieck vector representation.
159    * </p>
