17    /**
18    *
19    * Decimal floating point library for Java
20    *
21    * <p>Another floating point class. This one is built using radix 10000
22    * which is 10<sup>4</sup>, so its almost decimal.</p>
23    *
24    * <p>The design goals here are:
25    * <ol>
26    * <li>Decimal math, or close to it</li>
27    * <li>Settable precision (but no mix between numbers using different settings)</li>
28    * <li>Portability. Code should be keep as portable as possible.</li>
29    * <li>Performance</li>
30    * <li>Accuracy - Results should always be +/- 1 ULP for basic
31    * algebraic operation</li>
32    * <li>Comply with IEEE 854-1987 as much as possible.
33    * (See IEEE 854-1987 notes below)</li>
34    * </ol></p>
35    *
36    * <p>Trade offs:
37    * <ol>
38    * <li>Memory foot print. I'm using more memory than necessary to
39    * represent numbers to get better performance.</li>
40    * <li>Digits are bigger, so rounding is a greater loss. So, if you
41    * really need 12 decimal digits, better use 4 base 10000 digits
42    * there can be one partially filled.</li>
43    * </ol></p>
44    *
45    * <p>Numbers are represented in the following form:
46    * <pre>
47    * n = sign &times; mant &times; (radix)<sup>exp</sup>;</p>
48    * </pre>
49    * where sign is &plusmn;1, mantissa represents a fractional number between
50    * zero and one. mant[0] is the least significant digit.
51    * exp is in the range of -32767 to 32768</p>
52    *
53    * <p>IEEE 854-1987 Notes and differences</p>
54    *
55    * <p>IEEE 854 requires the radix to be either 2 or 10. The radix here is
56    * 10000, so that requirement is not met, but it is possible that a
57    * subclassed can be made to make it behave as a radix 10
58    * number. It is my opinion that if it looks and behaves as a radix
59    * 10 number then it is one and that requirement would be met.</p>
60    *
61    * <p>The radix of 10000 was chosen because it should be faster to operate
62    * on 4 decimal digits at once instead of one at a time. Radix 10 behavior
63    * can be realized by add an additional rounding step to ensure that
64    * the number of decimal digits represented is constant.</p>
65    *
66    * <p>The IEEE standard specifically leaves out internal data encoding,
67    * so it is reasonable to conclude that such a subclass of this radix
68    * 10000 system is merely an encoding of a radix 10 system.</p>
69    *
70    * <p>IEEE 854 also specifies the existence of "sub-normal" numbers. This
71    * class does not contain any such entities. The most significant radix
72    * 10000 digit is always non-zero. Instead, we support "gradual underflow"
73    * by raising the underflow flag for numbers less with exponent less than
74    * expMin, but don't flush to zero until the exponent reaches MIN_EXP-digits.
75    * Thus the smallest number we can represent would be:
76    * 1E(-(MIN_EXP-digits-1)&lowast;4), eg, for digits=5, MIN_EXP=-32767, that would
77    * be 1e-131092.</p>
78    *
79    * <p>IEEE 854 defines that the implied radix point lies just to the right
80    * of the most significant digit and to the left of the remaining digits.
81    * This implementation puts the implied radix point to the left of all
82    * digits including the most significant one. The most significant digit
83    * here is the one just to the right of the radix point. This is a fine
84    * detail and is really only a matter of definition. Any side effects of
85    * this can be rendered invisible by a subclass.</p>
86    *
87    */
