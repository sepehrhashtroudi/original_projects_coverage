31    /**
32    * Beginnings of an optimization to specialize the initial module at the cost of
33    * increasing code in later modules. This is still very experimental.
34    *
35    * High-level overview:
36    *
37    * This optimization replaces functions in the initial module with specialized
38    * versions that are only safe in the initial module. The original, general,
39    * versions of the functions are "fixed up" in later modules. This optimization
40    * can shrink the initial module significantly but the fixup code in later
41    * modules increases overall code size.
42    *
43    * Implementation approach:
44    *
45    * We take a ridiculously naive approach: remove the initial module
46    * from the rest of the AST, optimize it with existing optimization passes
47    * (recording which functions have been specialized), put it back in the AST,
48    * and add fixups restoring the general versions of the functions in each module
49    * that depends on the initial module.
50    *
51    * Since it is only safe to specialize functions that can be fixed up, we
52    * don't allow specialization of local functions and functions that
53    * are aliased.
54    *
55    * We currently run three optimizations on the isolated AST: InlineFunctions,
56    * DevirtualizePrototypeMethods, and RemoveUnusedPrototypeProperties.
57    *
58    * These optimizations rely on a coarse-grained name-based analysis to
59    * maintain safety properties and thus are likely to see some benefit when
60    * applied in isolation.
61    *
62    * InlineFunctions is truly specializing -- it replaces functions with
63    * versions that have calls to other functions inlined into them, while
64    * RemoveUnusedPrototypeProperties is really just removing properties that
65    * aren't used in the initial module and adding copies further down in the
66    * module graph. It would probably be more elegant to give
67    * CrossModuleMethodMotion permission to make copies of methods instead.
68    *
69    * There are additional passes that might benefit from being made
70    * specialization-aware:
71    *
72    * - OptimizeParameters
73    *
74    * - Any pass that is too slow to run over the entire AST but might
75    * be acceptable on only the initial module:
76    * - RemoveUnusedNames
77    *
78    * - Also, any pass that uses the results of PureFunctionIdentifier to
79    * determine when it is safe to remove code might benefit (e.g. the peephole
80    * passes), since PureFunctionIdentifier relies on SimpleDefinitionFinder,
81    * which would be more precise when running on only the initial module.
82    *
83    * @author dcc@google.com (Devin Coughlin)
84    */
