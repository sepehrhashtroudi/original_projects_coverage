26    /**
27    * <p>
28    * A specialized {@link BackgroundInitializer} implementation that can deal with
29    * multiple background initialization tasks.
30    * </p>
31    * <p>
32    * This class has a similar purpose as {@link BackgroundInitializer}. However,
33    * it is not limited to a single background initialization task. Rather it
34    * manages an arbitrary number of {@code BackgroundInitializer} objects,
35    * executes them, and waits until they are completely initialized. This is
36    * useful for applications that have to perform multiple initialization tasks
37    * that can run in parallel (i.e. that do not depend on each other). This class
38    * takes care about the management of an {@code ExecutorService} and shares it
39    * with the {@code BackgroundInitializer} objects it is responsible for; so the
40    * using application need not bother with these details.
41    * </p>
42    * <p>
43    * The typical usage scenario for {@code MultiBackgroundInitializer} is as
44    * follows:
45    * <ul>
46    * <li>Create a new instance of the class. Optionally pass in a pre-configured
47    * {@code ExecutorService}. Alternatively {@code MultiBackgroundInitializer} can
48    * create a temporary {@code ExecutorService} and delete it after initialization
49    * is complete.</li>
50    * <li>Create specialized {@link BackgroundInitializer} objects for the
51    * initialization tasks to be performed and add them to the {@code
52    * MultiBackgroundInitializer} using the
53    * {@link #addInitializer(String, BackgroundInitializer)} method.</li>
54    * <li>After all initializers have been added, call the {@link #start()} method.
55    * </li>
56    * <li>When access to the result objects produced by the {@code
57    * BackgroundInitializer} objects is needed call the {@link #get()} method. The
58    * object returned here provides access to all result objects created during
59    * initialization. It also stores information about exceptions that have
60    * occurred.</li>
61    * </ul>
62    * </p>
63    * <p>
64    * {@code MultiBackgroundInitializer} starts a special controller task that
65    * starts all {@code BackgroundInitializer} objects added to the instance.
66    * Before the an initializer is started it is checked whether this initializer
67    * already has an {@code ExecutorService} set. If this is the case, this {@code
68    * ExecutorService} is used for running the background task. Otherwise the
69    * current {@code ExecutorService} of this {@code MultiBackgroundInitializer} is
70    * shared with the initializer.
71    * </p>
72    * <p>
73    * The easiest way of using this class is to let it deal with the management of
74    * an {@code ExecutorService} itself: If no external {@code ExecutorService} is
75    * provided, the class creates a temporary {@code ExecutorService} (that is
76    * capable of executing all background tasks in parallel) and destroys it at the
77    * end of background processing.
78    * </p>
79    * <p>
80    * Alternatively an external {@code ExecutorService} can be provided - either at
81    * construction time or later by calling the
82    * {@link #setExternalExecutor(ExecutorService)} method. In this case all
83    * background tasks are scheduled at this external {@code ExecutorService}.
84    * <strong>Important note:</strong> When using an external {@code
85    * ExecutorService} be sure that the number of threads managed by the service is
86    * large enough. Otherwise a deadlock can happen! This is the case in the
87    * following scenario: {@code MultiBackgroundInitializer} starts a task that
88    * starts all registered {@code BackgroundInitializer} objects and waits for
89    * their completion. If for instance a single threaded {@code ExecutorService}
90    * is used, none of the background tasks can be executed, and the task created
91    * by {@code MultiBackgroundInitializer} waits forever.
92    * </p>
93    *
94    * @since 3.0
95    * @version $Id$
96    */
